# Coolify, GitHub App, and Cloudflare Tunnel

This document describes the network path required for a self-hosted Coolify
instance to receive GitHub App callbacks when the server does not have a
publicly reachable address.

## Architecture

```text
GitHub
  |
  | HTTPS webhook / OAuth callback
  v
Cloudflare public hostname
  |
  | outbound Cloudflare Tunnel
  v
cloudflared on the Coolify host
  |-- http://localhost:8000  Coolify dashboard
  `-- http://localhost:6001  Coolify realtime service
```

The tunnel exposes the Coolify control plane. It does not automatically expose
every application deployed by Coolify. Application hostnames need their own
Cloudflare routes to the Coolify reverse proxy, or another deliberate ingress
design.

The Coolify dashboard hostname must also be reachable by GitHub Actions when
using the repository's release-promotion workflow. Store that public base URL
as the `COOLIFY_URL` GitHub Environment secret; do not use an internal
`localhost` address because GitHub-hosted runners cannot reach it.

## Prerequisites

- The domain is authoritative in Cloudflare.
- The server can make outbound connections to Cloudflare (including port
  `7844`).
- Coolify is installed and reachable locally on port `8000`.
- Docker Compose v2 is installed on the host. Verify with:

  ```sh
  docker compose version
  ```

If this prints `docker: unknown command: docker compose`, the Docker Compose
plugin is missing. The Coolify installer can report this as a misleading
“Failed to parse docker-compose configuration” error. Install the plugin from
Docker's official APT repository, then rerun the check:

```sh
apt update
apt install -y docker-compose-plugin
docker compose version
```

On Ubuntu, the Docker repository must be configured before that package can be
found. Do not replace the Coolify Compose files while diagnosing this error.

## Create the tunnel

1. In Cloudflare, open **Zero Trust > Networks > Tunnels** and create a
   remotely managed tunnel.
2. Select **Cloudflared**, choose Linux, and run the installation command
   generated by Cloudflare on the Coolify host. It resembles:

   ```sh
   cloudflared service install <TUNNEL_TOKEN>
   ```

3. Wait for the tunnel to show **Healthy**.

Cloudflare creates the DNS CNAME automatically when a published application
route is added. Do not also create an A record for the same tunnel hostname.

## Publish the Coolify control plane

Add these published application routes to the tunnel:

| Public hostname | Local service | Purpose |
| --- | --- | --- |
| `coolify.example.com` | `http://localhost:8000` | Dashboard, GitHub App callbacks, webhooks |
| `coolify-realtime.example.com` | `http://localhost:6001` | Coolify realtime/WebSocket traffic |

Replace `example.com` with the real Cloudflare zone. Test the dashboard from
an external network:

```sh
curl -I https://coolify.example.com
```

Do not put a Cloudflare Access login in front of the dashboard hostname while
registering the GitHub App. GitHub's webhook requests cannot complete an
interactive Access login. Coolify's own authentication remains in effect.

## Configure Coolify

In the Coolify dashboard:

1. Open **Settings**.
2. Set **Instance Domain** to `https://coolify.example.com`.
3. Save the setting.
4. Start the server proxy if it is not already running.

On the host, set the realtime hostname in
`/data/coolify/source/.env`:

```env
PUSHER_HOST=coolify-realtime.example.com
PUSHER_PORT=443
```

Do not include `https://` in `PUSHER_HOST`. Apply the change by rerunning the
Coolify installer/update command:

```sh
curl -fsSL https://cdn.coollabs.io/coolify/install.sh | bash
```

Verify the dashboard and realtime connection before configuring GitHub.

## Configure GitHub App

After `https://coolify.example.com` works publicly:

1. In Coolify, open **Sources > Add > GitHub App**.
2. Use the automated registration flow.
3. Create the app when redirected to GitHub.
4. Install it only for the repositories that Coolify should access.
5. Return to Coolify and select the app for private repository resources.

The webhook URL must use the Coolify instance domain and have this shape:

```text
https://coolify.example.com/webhooks/source/github/events
```

Use GitHub's **Recent Deliveries** view to confirm webhook responses after
installation. A timeout or `502` indicates the tunnel route or local service;
an HTTP `401`/`403` usually indicates Cloudflare Access or another security
layer is blocking GitHub.

## Application ingress through the tunnel

The dashboard tunnel is not an ingress route for Zongo. If applications also
need to be reachable through Cloudflare Tunnel, publish their hostnames to the
Coolify reverse proxy (normally `http://localhost:80` or `https://localhost:443`)
and keep the Compose services themselves without host-published ports.

For this project, keep `ADMIN_HOSTNAME` private to the Tailscale network unless
the access model is intentionally changed. Do not put the admin hostname on a
public Cloudflare route without revisiting the VPN middleware and security
assumptions.

## Security and operations

- Keep Postgres, Redis, and Docker ports private.
- Restrict SSH to trusted addresses or a private network.
- Store Cloudflare tunnel tokens, Coolify secrets, GitHub private keys, and
  webhook secrets in secret storage; never commit them.
- If using a direct public ingress instead of a tunnel, open only the ports
  required by the chosen design. With a tunnel, inbound `80`/`443` are not
  required for the tunnel itself.
- Record the tunnel hostname and the Coolify instance domain in the
  environment's release notes.
